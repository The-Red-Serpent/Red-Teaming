## Model Context Protocol

Model Context Protocol is an open standard protocol introduced by Anthropic in November 2024 that defines a standardized way for AI applications to connect to external tools, data sources, and servicesthrough structured API calls. It establishes a common communication interface  based on JSON-RPC 2.0  so any AI application can discover and interact with any external system through three primitives: tools (functions the LLM can call to take actions), resources (data the LLM can read), and prompts (pre-built templates).

Before MCP every AI application had to build custom integrations for every external system — a different connector for GitHub, databases, file systems  none reusable across applications. MCP solves this with one protocol that any host implements once to connect to any MCP-compatible server, and any server implements once to become accessible to any host. Think of it as USB-C for AI — one universal interface instead of a different connector for everything.

## MCP Host

The host is the AI application that the user actually interacts with. Claude Desktop, Cursor, VS Code with Copilot, a custom chatbot you built — these are hosts. The host contains the LLM and is responsible for the entire user experience. It decides which MCP servers to connect to, manages those connections, and presents the combined capabilities to the LLM. Think of the host as the boss — it is in charge of everything and orchestrates all the moving parts.

## MCP Client

The client lives inside the host and is the dedicated communication handler for one specific MCP server. Every time the host connects to a new server, it creates a new client instance just for that server. The client speaks the MCP protocol  it handles the handshake, discovers what tools the server offers, sends tool call requests, and receives results. The user never sees the client. It is purely internal plumbing. Think of the client as a translator sitting between the host and one specific server — it handles all the back and forth communication so the host does not have to worry about protocol details.

## MCP Server

The server is an external process that wraps a real-world system and makes it accessible to any AI application through the MCP protocol. A filesystem server wraps your file system. A GitHub server wraps the GitHub API. A PostgreSQL server wraps your database. The server exposes tools with names and descriptions so the LLM can understand what it can do. When a tool gets called the server actually executes it against the real system and returns the result. Think of the server as a skilled contractor — it has expertise in one specific external system, it advertises what it can do, and it does the actual work when asked.

## MCP Primitives
### Tools

Tools are functions that the LLM can call to take actions or get information from the outside world. Each tool has three things — a name, a description written in plain English so the LLM understands when to use it, and a parameter schema defining what inputs it needs. When the LLM decides to call a tool it generates a structured request with the tool name and arguments. The MCP client sends that to the server, the server executes it against the real system, and returns the result back to the LLM.

Tools are the most important primitive in MCP. They are what give the LLM the ability to actually do things — not just talk about things.

Examples:
```
read_file     → reads a file from the filesystem
write_file    → writes content to a file
query_db      → runs a SQL query against a database
search_web    → searches the internet
create_issue  → creates a GitHub issue
send_email    → sends an email
run_command   → executes a bash command
```

### Resources

Resources are data that the server exposes for the LLM to read. Unlike tools which execute actions, resources are passive — they just provide content. Each resource has a URI that identifies it, like a file path or a URL. The host can read resources and inject their content directly into the LLM's context window.

Think of resources as files you can open and read. Tools are things you can do. Resources are things you can see.

Examples:
```
file:///home/user/report.pdf     → a file on disk
db://customers/schema            → a database schema
github://repo/README.md          → a file in a repo
config://app/settings            → application config
```


### Prompts

Prompts are pre-built templates that the server exposes for common tasks. They are reusable prompt structures that the host or user can invoke to generate well-structured inputs for specific workflows.

Example  a GitHub MCP server might expose a prompt template called `code_review` that takes a PR number and generates a structured code review prompt. Instead of the user writing "please review PR #42 checking for security issues, performance problems, and style violations", they just invoke the `code_review` prompt with `pr=42` and the template handles the rest. Prompts are the least used of the three primitives. Tools and resources do the heavy lifting in most MCP implementations.

## MCP Communication

MCP Components Communicate
Communication in MCP is built on JSON-RPC 2.0, which provides a clear, predictable structure for requests, responses, and notifications.

MCP supports two main transport mechanisms:

    STDIO : for local integrations, where the server runs on the same device as the client.
    HTTP + SSE (Server-Sent Events) : for remote deployments, allowing hosts and servers to be decoupled and run anywhere cloud, local, or hybrid setups.


Attack surface:
  Malicious server responses  → prompt injection
  Tool description poisoning  → pre-execution manipulation
  Typosquatting servers       → supply chain
  Excessive permissions       → blast radius amplification
  Cross-server injection      → lateral movement between tools
  Rug pull                    → post-adoption compromise
