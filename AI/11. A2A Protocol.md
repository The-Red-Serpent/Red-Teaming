### A2A Protocol
Agent-to-Agent (A2A) protocol is a standardized set of rules, formats, and procedures that govern how autonomous AI agents communicate, exchange data, and coordinate actions with each other in a multi-agent system.

### A2A Client
An application or agent that initiates requests to an A2A server on behalf of a user or another system. The client does not need to know anything about how the remote agent works internally — it just sends tasks and receives results. In a multi-agent system the orchestrator agent acts as the A2A client when delegating work to specialist agents.

### A2A Server (Remote Agent)
An AI agent or agentic system that exposes an HTTP endpoint implementing the A2A protocol. It receives tasks from clients, processes them using whatever internal tools, memory, and logic it has, and returns results or status updates. Critically the server is a black box from the client's perspective — its internal architecture, tools, memory, and reasoning are completely hidden. The client only sees inputs and outputs.

### Agent Card
A JSON metadata document that every A2A server publishes at a well-known URL. It is the agent's identity document — it tells any client what the agent can do, how to reach it, and how to authenticate with it.

```json
{
  "name": "Data Analysis Agent",
  "description": "Analyzes datasets and produces statistical summaries",
  "url": "https://agent.example.com/a2a",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true
  },
  "skills": [
    {
      "id": "analyze_csv",
      "name": "Analyze CSV",
      "description": "Takes a CSV file and returns statistical analysis",
      "inputModes": ["file"],
      "outputModes": ["text", "data"]
    }
  ],
  "authentication": {
    "schemes": ["bearer"]
  }
}
```

The agent card is how agents discover each other. An orchestrator reads agent cards to understand what specialist agents are available and what they can do — then decides which one to delegate a task to.

### Task
The fundamental unit of work in A2A. Every interaction is framed as a task with a unique ID. Tasks are stateful  they progress through a defined lifecycle:

```
submitted → working → (input-required) → completed
                                       → failed
                                       → canceled
```

Tasks persist on the server side. The client can check the status of a task at any time by its ID. Long-running tasks continue processing even if the client disconnects.

### Message
A single communication turn between client and agent, equivalent to one message in a conversation. Every message has a role — either "user" (from the client) or "agent" (from the server) — and contains one or more parts. Messages form the conversation history of a task.

### Part
The smallest unit of content inside a message or artifact. Three types:

**TextPart** — plain text content. A question, an instruction, a response.

**FilePart** — a file, either as raw bytes (inline) or as a URI pointing to the file location. Used to send documents, images, CSVs, or any file to the agent.

**DataPart** — structured JSON data. Used when the content has a defined schema — form submissions, API responses, database records.

A single message can contain multiple parts of different types — for example a TextPart with instructions plus a FilePart with the document to analyze.

### Artifact
The output produced by the agent as a result of completing a task. Like messages, artifacts are composed of parts. The difference is that artifacts are the final deliverables — the thing the agent produced — while messages are the conversation turns that led to it.

Examples: a generated report (TextPart), an analyzed image (FilePart), a structured data result (DataPart), a generated chart (FilePart).

Artifacts are versioned and indexed — an agent can produce multiple artifacts for a single task, each with its own index number. Artifacts can also be streamed incrementally — the agent appends chunks as it generates them rather than waiting until the entire artifact is complete.

### Streaming (SSE)
Server-Sent Events based real-time updates for long-running tasks. Instead of the client polling repeatedly for status, the server pushes incremental updates as they happen — status changes, artifact chunks, intermediate results. The client opens a persistent HTTP connection and the server sends events down it as the task progresses.

```
Client: POST /tasks/send (with stream=true)
Server: → event: task_status (working)
        → event: artifact_chunk (first 200 chars of report)
        → event: artifact_chunk (next 200 chars)
        → event: artifact_chunk (final chars)
        → event: task_status (completed)
```

Used for tasks where the user needs to see progress in real time — generating a long document, running a complex analysis, processing a large file.

### Push Notifications
Asynchronous task updates delivered via server-initiated HTTP POST requests to a webhook URL provided by the client. Used for long-running or disconnected scenarios where the client cannot maintain a persistent connection.

The client submits a task and provides a webhook URL. The server processes the task — potentially over minutes or hours — and POSTs status updates and results to the webhook as they occur. The client does not need to maintain any connection or poll for updates.

```
Client: POST /tasks/send
        { webhookUrl: "https://client.example.com/webhook", ... }
Server: [processes task for 10 minutes]
Server: POST https://client.example.com/webhook
        { taskId: "abc123", status: "completed", artifacts: [...] }
```

### Request/Response (Polling)
The simplest interaction pattern. Client sends a request, server responds synchronously if the task completes quickly. For longer tasks the server returns immediately with status "working" and the client periodically sends GET requests with the task ID to check progress until the task completes.

```
Client: POST /tasks/send → { taskId: "abc123", status: "working" }
Client: GET /tasks/abc123 → { status: "working" }
Client: GET /tasks/abc123 → { status: "working" }
Client: GET /tasks/abc123 → { status: "completed", artifacts: [...] }
```

Simple but inefficient for long tasks — the client has to keep checking.

### Context
An optional server-generated identifier used to logically group related tasks together. Tasks sharing a context ID are treated as part of the same logical session — the server can use context to maintain state and memory across multiple related tasks without the client having to resend all prior information.

Think of context as a session ID. Multiple tasks within the same context share a conversation history and accumulated state on the server side.

### Extension
A mechanism for agents to provide additional functionality beyond the core A2A specification. Extensions allow agents to advertise and use capabilities that are not part of the standard protocol — domain-specific features, custom authentication schemes, specialized data formats. Extensions are declared in the agent card so clients know what extended capabilities are available.
