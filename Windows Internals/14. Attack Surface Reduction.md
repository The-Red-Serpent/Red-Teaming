## Attack Surface Reduction
Attack Surface Reduction is a component of Microsoft Defender for Endpoint (formerly Windows Defender ATP) that applies rule-based behavioral controls to block common malware techniques. It is part of the broader Exploit Protection suite, alongside Controlled Folder Access and Network Protection. ASR is not signature-based. It uses built-in behavioral sensors in Windows to detect and block specific attack patterns at the point of execution.

ASR rules are enforced by the Microsoft Defender Antivirus service (MsMpEng.exe) via:

| Component | Role |
|-----------|------|
| MsMpEng.exe | Core antimalware service — evaluates ASR rules |
| mpengine.dll | Scanning engine — pattern matching and behavioral analysis |
| NisSrv.exe | Network Inspection Service — URL/domain reputation |
| Kernel callbacks | PsSetCreateProcessNotifyRoutine, PsSetCreateThreadNotifyRoutine, PsSetLoadImageNotifyRoutine |
| ETW (Event Tracing for Windows) | Telemetry collection for rule evaluation |

When a process attempts to:
* Create a child process\
* Load a DLL\
* Execute a script\
* Write to disk\
* Make a network connection\

The kernel callback fires, MsMpEng.exe evaluates the action against enabled ASR rules, and either blocks or allows it.

## ASR Status and Enumeration
Checking if ASR Is Enabled
#### Method 1: PowerShell (Most Reliable)

```
# Check if ASR rules are configured
Get-MpPreference | Select-Object -ExpandProperty AttackSurfaceReductionRules_Ids

# Check the mode of each rule
Get-MpPreference | Select-Object -ExpandProperty AttackSurfaceReductionRules_Actions
```


Output interpretation:
Empty result = No ASR rules configured (ASR is effectively off)\
List of GUIDs with 1 = Block mode\
List of GUIDs with 2 = Audit mode\
List of GUIDs with 3 = Warn mode\



## ASR Exclusions

ASR exclusions are paths, file hashes, or processes that you tell Microsoft Defender to **skip** when evaluating ASR rules. If a file or process matches an exclusion, ASR will allow the action even if it would normally be blocked by a rule.

### Types of Exclusions

| Type | What It Excludes | Example |
|------|------------------|---------|
| **Path exclusion** | All files under a specific folder | `C:\Tools\` |
| **File extension exclusion** | All files with a specific extension | `*.ps1` |
| **Process exclusion** | Actions performed by a specific executable | `C:\Program Files\MyApp\updater.exe` |
| **Hash exclusion** | A specific file by its SHA256 hash | `a1b2c3d4...` |

---

### Where Exclusions Are Configured

```powershell
# View current ASR exclusions
Get-MpPreference | Select-Object -ExpandProperty AttackSurfaceReductionOnlyExclusions

# View process exclusions (affects all Defender components)
Get-MpPreference | Select-Object -ExpandProperty ExclusionProcess
```

**Registry location**:
```
HKLM\SOFTWARE\Policies\Microsoft\Windows Defender\Windows Defender Exploit Guard\ASR\Exclusions
```

Bypass approaches:
* Use DLL sideloading via Office (Office loads a legitimate DLL that loads your malicious DLL)
* Use COM hijacking — Office can instantiate COM objects that execute code outside the Office process
* Use XLL add-ins (Excel) — these are DLLs loaded by Excel, not child processes
* Use VBA to call .NET via System.Diagnostics.Process (some rules don't block .NET-internal process creation)
